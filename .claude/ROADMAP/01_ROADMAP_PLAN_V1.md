# Financial Chart Analysis Platform — Implementation Roadmap

## Software Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| **Backend API** | Python 3.12 + FastAPI | Async, auto-generated OpenAPI spec, Pydantic validation, best-in-class data science ecosystem |
| **Frontend** | TypeScript + Next.js 14 (App Router) | SSR, API route proxying, React ecosystem; TailwindCSS + shadcn/ui for dashboard UI |
| **Charts** | TradingView Lightweight Charts | Open-source, ~35KB, native TS types, plugin architecture for custom overlays |
| **Data Analysis** | pandas + pandas-ta + mplfinance | Vectorized OHLCV ops, 130+ built-in indicators including `cdl_inside()`, chart image generation |
| **AI Inference** | Together AI (Llama 4 Maverick) | Vision model for chart pattern confirmation; OpenAI-compatible API; ~$0.002/chart |
| **Database** | PostgreSQL + TimescaleDB | Time-series hypertables, continuous aggregates, 90%+ compression on OHLCV data |
| **Cache / Queue** | Redis + Celery | Rate-limit management, API response caching, scheduled data ingestion jobs |
| **Market Data** | Finnhub (dev) → Polygon.io (prod) | 60 req/min free tier; institutional-grade OHLCV in production |
| **Fundamentals** | Financial Modeling Prep (FMP) | SEC filings, earnings transcripts, 30+ years of financials — from $22/mo |
| **Type Safety** | Orval (OpenAPI → TS client) | Auto-generates typed TypeScript hooks from FastAPI's OpenAPI spec — zero manual type sync |
| **Containerization** | Docker + docker-compose | Reproducible local dev: FastAPI + Next.js + TimescaleDB + Redis in one command |

```sql
CREATE TABLE ohlcv (
    time        TIMESTAMPTZ NOT NULL,
    ticker      TEXT NOT NULL,
    open        NUMERIC,
    high        NUMERIC,
    low         NUMERIC,
    close       NUMERIC,
    volume      BIGINT
);
SELECT create_hypertable('ohlcv', 'time');
CREATE INDEX idx_ohlcv_ticker_time ON ohlcv (ticker, time DESC);

-- Continuous aggregate for weekly candles (useful later)
CREATE MATERIALIZED VIEW ohlcv_weekly
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 week', time) AS week,
       ticker,
       first(open, time) AS open,
       max(high) AS high,
       min(low) AS low,
       last(close, time) AS close,
       sum(volume) AS volume
FROM ohlcv
GROUP BY week, ticker;
```

### Week 2: Inside Day Detection Engine

**Goal:** Core detection algorithm working, exposed via API.

**Tasks:**
1. **Prototype in Jupyter first** (`notebooks/01_inside_day_detection.ipynb`). Load OHLCV from TimescaleDB into pandas, iterate on the detection logic visually before committing to the module.

2. Build `modules/chart_patterns/detector.py` with both detection modes:

```python
import pandas as pd
from dataclasses import dataclass

@dataclass
class InsideDayPattern:
    ticker: str
    mother_bar_date: str
    consecutive_count: int       # 2, 3, 4, etc.
    mother_high: float
    mother_low: float
    compression_ratio: float     # range narrowing %
    volume_declining: bool       # volume confirmation
    is_nr4: bool                 # narrowest range in 4 days
    detection_mode: str          # "nested" or "mother_bar"

def detect_inside_days(df: pd.DataFrame, mode: str = "nested") -> list[InsideDayPattern]:
    """
    Detect consecutive inside days from OHLCV DataFrame.

    Two modes:
    - "nested": each bar must be inside the PREVIOUS bar (coiling)
    - "mother_bar": all bars must be inside the FIRST bar's range
    """
    df = df.sort_values("time").reset_index(drop=True)
    df["range"] = df["high"] - df["low"]

    if mode == "nested":
        df["inside"] = (df["high"] <= df["high"].shift(1)) & \
                        (df["low"] >= df["low"].shift(1))
    else:  # mother_bar mode
        # More complex: need to track the mother bar
        df["inside"] = False
        mother_idx = None
        for i in range(1, len(df)):
            if mother_idx is not None:
                if df.loc[i, "high"] <= df.loc[mother_idx, "high"] and \
                   df.loc[i, "low"] >= df.loc[mother_idx, "low"]:
                    df.loc[i, "inside"] = True
                    continue
            mother_idx = i  # Reset mother bar
            df.loc[i, "inside"] = False

    # Group consecutive inside days
    df["group"] = (~df["inside"]).cumsum()
    patterns = []
    for _, group in df[df["inside"]].groupby("group"):
        count = len(group)
        if count < 1:
            continue
        mother_idx = group.index[0] - 1
        if mother_idx < 0:
            continue

        mother = df.loc[mother_idx]
        last_bar = group.iloc[-1]
        patterns.append(InsideDayPattern(
            ticker=df.loc[mother_idx, "ticker"] if "ticker" in df.columns else "",
            mother_bar_date=str(mother["time"]),
            consecutive_count=count + 1,  # include mother bar
            mother_high=float(mother["high"]),
            mother_low=float(mother["low"]),
            compression_ratio=float(last_bar["range"] / mother["range"]),
            volume_declining=bool(group["volume"].is_monotonic_decreasing),
            is_nr4=bool(last_bar["range"] == df["range"].iloc[
                max(0, last_bar.name - 3):last_bar.name + 1
            ].min()),
            detection_mode=mode,
        ))
    return patterns
```

**Tasks:**

## Key Principles Throughout

**Start using it immediately.** The Phase 1 MVP should be something you check every morning. Real usage reveals what to prioritize next far better than a roadmap.

**One module at a time.** Each phase adds exactly one new folder under `modules/`. The `base_module.py` interface ensures they all plug in identically. Resist the urge to build infrastructure for features you haven't started yet.

**Prototype in notebooks, promote to modules.** Every detection algorithm starts in `notebooks/`, gets validated visually, then moves to `modules/` with tests. This is the fastest iteration cycle.

**Let Orval do the boring work.** Every time you add or change a FastAPI endpoint, re-run Orval. You should never be manually writing TypeScript types that mirror Python schemas.

**Cache aggressively.** Market data doesn't change after market close. Once the nightly ingestion runs, all pattern scans should read from TimescaleDB (not live API calls). Redis caches API responses for the frontend. This keeps Finnhub rate limits irrelevant for daily use.
